
# Pantheon OS Constitution

## Purpose

Pantheon OS exists to make Prompt Engines legible, controllable, and compounding without turning the lab into bureaucracy.

## Core operating laws

1. A.I. sets direction and retains final approval on meaningful decisions.
2. Agents execute bounded work against explicit goals, tasks, approvals, and budgets.
3. Skills are first-class operational assets, not incidental prompts.
4. Every skill must begin from concrete use cases and expose progressive disclosure:
   - frontmatter for triggering
   - `SKILL.md` for execution
   - linked templates, references, scripts, or assets only when needed
5. Skills are composable across agents and sit above tools/connectors as the recipe layer.
6. Every important skill is governed as living operational memory with owner, version, status, review cadence, and venture scope.
7. Every venture must expose owner, stage, status, health, next action, heartbeat, and budget posture.
8. Higher-risk actions require explicit governance gates before deployment, publication, spend escalation, or external communication.
9. Recurring loops must be explicit heartbeats with cadence, owner, expected output, and runtime budget.
10. Pantheon OS stays local-first. Hosted services are used only where natural and defensible.

## System layers

- Constitution aligns agents.
- Skills operationalize repeatable work.
- Memory preserves what is learned.
- Connectors provide access.

## Paperclip adoption stance

Paperclip is a strong reference system. Pantheon OS borrows useful patterns from it:
- goal and task hierarchy
- heartbeat and recurring check-in logic
- governance and approvals
- budget and cost visibility

Pantheon OS does not adopt Paperclip as its core substrate in v1.
