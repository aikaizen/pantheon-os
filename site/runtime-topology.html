PantheonOS separates named agents from the runtime substrate they occupy. The canonical rule is explicit: every Hermes instance is deployed in a single Docker container and supervised by a host-level overseer agent.

Canonical terms

Host overseer

The overseer agent runs on the host machine, outside the persona runtimes. It supervises Docker lifecycle, routing, runtime health, and restart actions.

Canonical terms

Hermes instance

A Hermes instance is the orchestration runtime. It provides memory, tools, delegation, messaging, scheduling, and execution harnessing for the agents inside it.

Canonical terms

Deployment unit

One Hermes instance equals one Docker container. The container is the runtime boundary. Agents are tenants inside that single supervised runtime.

Canonical terms

Agent slot

Each Hermes instance supports up to four agent slots. A slot can be active with a named agent or left available for future occupancy.

Why this matters

This model prevents a common confusion: one container per agent is not the PantheonOS default. The system is built around a supervised runtime that can host multiple named agents while remaining observable and operable as a single deployment unit.
