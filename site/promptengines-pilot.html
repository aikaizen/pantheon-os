Reference deployment

PromptEngines Pilot Readiness

PantheonOS should become useful to other operators by first becoming genuinely useful for PromptEngines.com and the PromptEngines suite of ventures/apps. This plan defines what the repo must contain before internal testing is real instead of theatrical.

Definition of ready

Ready means

PantheonOS can observe a real PromptEngines deployment with persona agents, controllers, runtimes, and channels; show them on one surface; and support internal operator usage without demanding a rebuild of existing agent setups.

Not ready means

The repo still treats PromptEngines as prose only, still lacks deployment bindings, or still shows mainly mock concepts without enough live observability to support daily use.

Required workstreams

1. Reference deployment model

Add explicit registries for runtimes, controllers, channels, and bindings. PromptEngines must be representable as a deployment topology, not just a venture list.

2. State engine v3

Observe persona status, controller status, runtime status, channel summaries, and bindings. Keep the engine observational.

3. Operating Terminal MVP

Make the terminal useful for real daily use: persona chat panes, controller panes, runtime health, artifacts, and interventions.

4. PromptEngines venture integration

Make the dashboard and terminal show which PromptEngines ventures are actually participating in the pilot and at what level.

5. Onboarding modes

Support Connect Existing Setup first, then Start Fresh. PantheonOS must adopt, not replace, existing agent estates.

6. Productization later

Once the pilot works internally, package PromptEngines as the first reusable reference deployment for outside operators.

Pilot gates

Before internal testing counts as real, the repo should have:

  • a declared PromptEngines deployment topology
  • registries for persona agents, runtimes, controllers, channels, and bindings
  • a state engine that can emit partial real observation
  • a terminal model that distinguishes persona panes from controller panes
  • observe / message / control permission modes
  • docs for Connect Existing Setup and Start Fresh
  • a written acceptance checklist for the pilot

Execution order

1. Lock the PromptEngines deployment model. 2. Add missing registry objects. 3. Expand the state engine. 4. Revise the terminal around real panes of relationship. 5. Add onboarding modes. 6. Wire Tier 1 ventures into the pilot. 7. Run the internal pilot before external packaging.
